# Scope Definition

Creates structured project boundaries — clearly defining what's in scope, what's out, and what's deferred — so everyone is on the same page before work begins.

## Questions It Answers

- "What exactly are we building (and what are we *not* building)?"
- "What assumptions are we making, and what happens if they're wrong?"
- "What are our constraints and how do they affect delivery?"
- "What are the key deliverables, and how will we know they're done?"

## Example Use Cases

- A Senior PM is about to kick off a payments integration and needs a clear scope document before the engineering team starts estimating work.
- After an ideation session generates 15 ideas, a product organization needs to draw boundaries around what makes the cut for v1 — with clear documentation for stakeholder alignment.
- A program sponsor wants to see assumptions and risks documented upfront, not discovered mid-sprint when it's too late to course-correct.

## What Makes It Special

- Import context directly from Ideation or Goal & Metric Planner sessions so you're not starting from scratch
- Drag-and-drop items between in-scope, out-of-scope, and deferred columns
- Every assumption lists what could go wrong and how to validate it
