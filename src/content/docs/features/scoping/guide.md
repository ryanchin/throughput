# How to Use Scope Definition

This guide walks you through creating clear project boundaries that keep your team aligned on what's in, what's out, and what's deferred.

## Step 1: Choose Your Starting Point

You can start from three sources:

- **Custom** — Enter a project name and product vision (minimum 50 characters) from scratch
- **Import from Ideation** — Pull in ideas from a completed Ideation Engine session
- **Import from OKRs** — Pull in objectives from a Goal & Metric Planner session

Importing pre-populates the form fields with relevant context.

### Examples of Great Input

**Project Name:**
- "Enterprise SSO & Role-Based Access Control"
- "Self-Serve Billing Portal v2"

**Product Vision (50+ characters):**
- "Enable enterprise IT administrators to manage user access, enforce SSO policies, and audit team activity across the platform — replacing the manual provisioning process that currently requires support tickets and 3-day turnaround."
- "Give finance teams the ability to manage subscriptions, view invoices, adjust seats, and export billing data without contacting sales — reducing billing-related support tickets by 60%."

**Constraints:**
- "Must integrate with Okta and Azure AD. Cannot change the existing user data model. Budget capped at 2 engineers for 6 weeks. SOC 2 audit in Q3 means security review is mandatory before launch."

**Stakeholder Needs:**
- "VP Sales needs the SSO feature as a checkbox item for enterprise RFPs. Customer Success needs admin controls to reduce provisioning support tickets. Security team needs audit logging for compliance."

## Step 2: Add Optional Context

Expand the optional fields to provide richer input:

- **Knowledge Bases** — Attach organizational docs for context-aware generation
- **Initial Requirements** — Any known requirements or constraints
- **Constraints** — Budget, timeline, technical, or regulatory limitations
- **Stakeholder Needs** — Who needs what from this project
- **Target Users** — Who the end users are

## Step 3: Review the Generated Scope

Product Studio generates:

- **Scope Statement** — A clear summary of the project boundaries
- **In-Scope Items** — What will be built
- **Out-of-Scope Items** — What explicitly will not be built
- **Deferred Items** — What may be built later but not now
- **Assumptions** — Each with a "risk if wrong" explanation
- **Constraints** — Categorized by type (technical, budget, timeline, regulatory, resource) with impact assessment
- **Deliverables** — With type classification and acceptance criteria

## Step 4: Organize and Edit

- **Drag and drop** items between in-scope, out-of-scope, and deferred columns — this is the primary way to reorganize scope
- **Add new items** to any category manually
- **Edit assumptions** to add validation approaches
- **Refine with AI** — Use global or item-specific revision requests

## Step 5: Export

Export as a PDF to share with stakeholders, project sponsors, or engineering leads before estimation begins.

## Tips

- Write a strong product vision (50+ characters) — this is the most important input
- Be explicit about what's out of scope — this prevents scope creep later
- For each assumption, make sure the "risk if wrong" is documented — this becomes your risk register
- Import from Ideation if you've already run an ideation session — it saves significant setup time
